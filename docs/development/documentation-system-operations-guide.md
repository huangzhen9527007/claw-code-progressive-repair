# 文档系统操作指南

## 概述

本文档详细说明了如何从记忆系统提取关键知识并整理到项目文档系统（docs/）中的完整流程、触发条件和最佳实践。

## 核心概念

### 文档系统 vs 记忆系统

| 方面 | 项目文档系统 (`docs/`) | 用户记忆系统 (`memory/`) |
|------|------------------------|--------------------------|
| **位置** | 项目代码目录中 | 用户配置目录中 |
| **内容** | 技术解决方案、架构设计、开发指南 | 个人工作记忆、经验记录 |
| **版本控制** | 是，由Git管理 | 否，个人存储 |
| **共享性** | 团队共享知识 | 仅个人使用 |
| **目的** | 项目技术参考 | 个人生产力增强 |

### 知识提炼过程

从记忆系统到项目文档的转化是一个"知识提炼"过程：
1. **记忆筛选** - 评估哪些经验具有普遍价值
2. **内容转化** - 将个人经验转化为团队共享文档
3. **质量保证** - 确保技术准确性和完整性

## 触发条件

### 应该触发文档创建的情况

✅ **具有普遍性** - 问题影响多个团队成员
✅ **经过验证** - 解决方案经过实际测试有效
✅ **提高效率** - 实践能提高开发效率或代码质量
✅ **架构变更** - 系统设计有重大变化
✅ **API更新** - 接口有新增或修改

### 不应该触发的情况

❌ **个人工作记录** - 仅个人使用的习惯记录
❌ **临时性解决方案** - 尚未验证的临时方案
❌ **未经验证的技术** - 技术方案未经实际测试
❌ **过于细节的代码** - 过于具体的实现细节

## 从记忆到文档的转化流程

### 步骤1：记忆筛选

**评估标准**:
1. **普遍性**: 是否对团队其他成员有用？
2. **完整性**: 是否有完整的解决方案？
3. **验证性**: 是否经过实际验证？
4. **时效性**: 是否长期有效？
5. **结构性**: 是否容易整理为结构化文档？

### 步骤2：文档定位

根据内容类型确定文档位置：

```bash
if [ 技术问题解决方案 ]; then
    docs/solutions/
elif [ 开发实践指南 ]; then
    docs/development/
elif [ 架构设计 ]; then
    docs/architecture/
elif [ API文档 ]; then
    docs/api/
elif [ 部署运维 ]; then
    docs/deployment/
elif [ 故障排除 ]; then
    docs/troubleshooting/
fi
```

### 步骤3：内容转化

**转化流程**: 原始记忆 → 提炼核心 → 结构化 → 补充细节 → 验证更新

**转化要点**:
1. **去除个人化表达** - 将"我"改为"开发者"，"我的项目"改为"本项目"
2. **补充背景信息** - 添加问题背景、适用场景、前置条件
3. **添加示例代码** - 提供可运行的代码示例
4. **结构化组织** - 使用清晰的标题层级和列表
5. **验证信息准确性** - 检查技术细节的正确性

### 步骤4：质量检查

**文档质量检查清单**:
- [ ] **技术准确性** - 所有技术信息经过验证
- [ ] **结构完整性** - 覆盖问题的各个方面
- [ ] **示例可运行** - 示例代码可以实际运行
- [ ] **链接有效** - 所有内部和外部链接有效
- [ ] **版本信息正确** - 包含正确的版本和日期信息
- [ ] **团队评审通过** - 经过团队其他成员评审

## 自动化辅助

### 文档生成辅助

使用脚本辅助文档创建:
1. 从记忆文件提取结构化信息
2. 生成文档模板
3. 检查文档完整性
4. 验证链接和引用

**命令示例**:
```bash
bun run scripts/generate-doc-from-memory.js \
  --memory windows_cmd_window_behavior_knowledge.md \
  --output docs/solutions/windows-cmd-window-behavior/README.md \
  --type solution
```

### 文档同步检查

定期检查文档与代码同步性:
1. API文档 vs 实际接口
2. 架构文档 vs 实际代码结构
3. 部署指南 vs 实际部署脚本
4. 解决方案 vs 实际代码实现

**检查脚本**:
```bash
bun run scripts/check-doc-sync.js \
  --docs docs/ \
  --code src/ \
  --report sync-report.md
```

## 最佳实践

### 文档创建时机

**应该触发文档创建的时机**:
1. 解决了团队中多人遇到的问题
2. 发现了重要的技术原理或限制
3. 建立了新的开发流程或规范
4. 系统架构有重大变更
5. API接口有新增或修改

**不应该触发的情况**:
1. 个人工作习惯记录
2. 临时性的解决方案
3. 尚未验证的技术方案
4. 过于细节的实现代码

### 文档质量保证

**确保文档质量的关键要素**:
1. **准确性** - 所有技术信息经过验证
2. **完整性** - 覆盖问题的各个方面
3. **可读性** - 结构清晰，语言简洁
4. **实用性** - 提供具体的实施步骤
5. **可维护性** - 易于更新和扩展

## 总结

### 触发原则

docs文档系统是**手动触发**的，基于价值判断和团队需求。触发决策需要考虑：
- 知识的普遍性和复用性
- 解决方案的验证程度
- 对团队的价值贡献
- 维护成本与收益比

### 关键触发点

1. ✅ **普遍性问题** - 影响多个团队成员
2. ✅ **验证解决方案** - 经过实际测试有效
3. ✅ **架构变更** - 系统设计有重大变化
4. ✅ **最佳实践** - 能提高开发效率或质量
5. ✅ **API更新** - 接口有新增或修改

### 避免触发

1. ❌ **个人工作记录** - 仅个人使用的经验
2. ❌ **临时性解决方案** - 尚未稳定的方案
3. ❌ **未经验证的技术** - 技术方案未经测试
4. ❌ **过于细节的代码** - 过于具体的实现
