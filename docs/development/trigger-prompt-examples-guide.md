# 文档系统触发提示词指南

## 概述

本文档提供了从记忆系统提取知识到项目文档系统的具体触发提示词示例、模板和工作流程。这些提示词可以帮助开发者系统化地将个人经验转化为团队共享的文档。

## 核心概念

### 触发原则
- **手动触发**：基于价值判断和团队需求
- **价值导向**：只整理具有普遍性、经过验证的知识
- **质量保证**：确保文档的准确性、完整性和实用性

### 知识提炼过程
从个人记忆到团队文档的转化需要：
1. **验证** - 技术方案经过实际测试
2. **结构化** - 组织成清晰的文档结构
3. **通用化** - 去除个人化表达，补充团队上下文

## 触发条件矩阵

### 应该触发文档创建的情况
| 条件 | 说明 | 示例 |
|------|------|------|
| ✅ **普遍性问题** | 影响多个团队成员 | Windows CMD窗口行为问题 |
| ✅ **验证解决方案** | 经过实际测试有效 | WeChat CLI编码问题解决方案 |
| ✅ **提高效率实践** | 能提高开发效率或代码质量 | PATH环境变量配置指南 |
| ✅ **架构重大变更** | 系统设计有重大变化 | LLM知识库架构模式 |
| ✅ **API接口更新** | 接口有新增或修改 | OpenAI兼容API适配器 |

### 不应该触发的情况
| 条件 | 说明 | 示例 |
|------|------|------|
| ❌ **个人工作记录** | 仅个人使用的习惯记录 | 个人调试笔记 |
| ❌ **临时性解决方案** | 尚未验证的临时方案 | 快速修复的临时代码 |
| ❌ **未经验证的技术** | 技术方案未经实际测试 | 未测试的新框架探索 |
| ❌ **过于细节的代码** | 过于具体的实现细节 | 单个函数的详细实现 |

## 实际用例示例

### 用例1：Windows CMD窗口行为知识

**记忆来源**: `windows_cmd_window_behavior_knowledge.md`
**触发提示词**:
```
基于记忆中的Windows CMD窗口行为关键知识，创建技术解决方案文档：

**文档位置**: `docs/solutions/windows-cmd-window-behavior/`
**文档标题**: Windows命令行窗口生命周期管理解决方案
**内容要点**:
1. 问题描述：双击BAT文件无法保持CMD窗口打开
2. 技术原理：临时窗口 vs 持久窗口的区别
3. 解决方案：正确的工作流程和最佳实践
4. 检测脚本：如何判断当前窗口类型
5. 常见误解：澄清关于pause命令和cmd /k的误解

**价值评估**:
- ✅ 具有普遍性：所有Windows开发者都会遇到
- ✅ 经过验证：实际测试确认了技术原理
- ✅ 解决方案完整：提供了具体实施步骤
- ✅ 团队共享价值：避免团队成员重复踩坑
```

## 转化流程提示词

### 步骤1：记忆筛选

```
检查记忆文件，评估文档价值:

评估标准:
1. 普遍性: 是否对团队其他成员有用？
2. 完整性: 是否有完整的解决方案？
3. 验证性: 是否经过实际验证？
4. 时效性: 是否长期有效？
5. 结构性: 是否容易整理为结构化文档？

决策标准:
- 如果满足3个以上条件 → 应该整理为文档
- 如果满足2个条件 → 需要进一步评估
- 如果满足1个或以下条件 → 暂时不整理
```

### 步骤2：文档定位

```
确定文档应该放在哪个目录:

if [ 技术问题解决方案 ]; then
    docs/solutions/
elif [ 开发实践指南 ]; then
    docs/development/
elif [ 架构设计 ]; then
    docs/architecture/
elif [ API文档 ]; then
    docs/api/
elif [ 部署运维 ]; then
    docs/deployment/
elif [ 故障排除 ]; then
    docs/troubleshooting/
fi
```

### 步骤3：内容转化

```
将记忆内容转化为文档内容:
原始记忆 → 提炼核心 → 结构化 → 补充细节 → 验证更新

转化要点:
1. 去除个人化表达（将"我"改为"开发者"）
2. 补充背景信息（问题背景、适用场景）
3. 添加示例代码（可运行的代码示例）
4. 结构化组织（清晰的标题层级和列表）
5. 验证信息准确性（检查技术细节）
```

### 步骤4：质量检查

```
执行文档质量检查:
- [ ] 技术准确性: 所有技术信息经过验证
- [ ] 结构完整性: 覆盖问题的各个方面
- [ ] 示例可运行: 示例代码可以实际运行
- [ ] 链接有效: 所有内部和外部链接有效
- [ ] 版本信息正确: 包含正确的版本和日期信息
- [ ] 团队评审通过: 经过团队其他成员评审
```

## 最佳实践提示词

### 文档创建时机

```
当遇到以下情况时，应该触发文档创建:
1. 解决了团队中多人遇到的问题
2. 发现了重要的技术原理或限制
3. 建立了新的开发流程或规范
4. 系统架构有重大变更
5. API接口有新增或修改

不应该触发的情况:
1. 个人工作习惯记录
2. 临时性的解决方案
3. 尚未验证的技术方案
4. 过于细节的实现代码
```

### 文档价值评估

```
评估文档价值:
1. 普遍性: 是否对团队其他成员有用？
2. 完整性: 是否有完整的解决方案？
3. 验证性: 是否经过实际验证？
4. 时效性: 是否长期有效？
5. 结构性: 是否容易整理为结构化文档？

价值评分:
- 5分: 满足所有条件，高价值文档
- 3-4分: 满足大部分条件，中等价值
- 1-2分: 满足少数条件，低价值
- 0分: 不满足任何条件，不需要整理
```
